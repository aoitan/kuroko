# Red Team Review - Issue #22: ルールベースによるTODO/期限等の抽出

## 1. 対象
- `InferenceEngine` (正規表現による属性抽出エンジン)
- `inferences` テーブル (抽出結果の永続化層)
- `memo_collector.py` への統合処理

## 2. 前提
- LLMを使用せず、軽量な正規表現とキーワードマッチングで「候補」を抽出する。
- 抽出結果は後続の `shinko` (LLM層) や UI でのフィルタリングに利用される。
- 主に対象は日本語のメモ書き (`memo.md`) である。

## 3. 主要な懸念点一覧
1. **ReDoS (正規表現によるサービス拒否攻撃) の脆弱性**: 複雑な正規表現やネストされた繰り返しにより、特定の入力で CPU 負荷が 100% になり、処理がハングするリスク。
2. **相対的な日付表現の固定化リスク**: 「明日」「来週」などの相対表現を抽出時に単純な文字列として保存すると、後から参照した際に「いつの明日か」が不明になる。
3. **ノイズ（誤検知）による DB 肥大化**: 抽出ルールが緩すぎると、単なる日記や文章の断片が大量に TODO 候補として登録され、DB 容量を圧迫し、後続処理の精度を下げる。
4. **ルールのメンテナンス性と「秘伝のタレ」化**: 正規表現が複雑化し、特定の開発者しか修正できない「読みづらい正規表現」が蓄積される。
5. **ルールの更新と既存データの不整合**: 抽出ルールを改善・追加した際、過去にインポート済みのチャンクに対して再抽出を行わないと、新旧データで検索性に差が出る。

## 4. 各懸念の重大度 / 発生条件 / 影響

| 懸念点 | 重大度 | 発生条件 | 影響 |
| :--- | :--- | :--- | :--- |
| **ReDoS** | **高** | 悪意のある、または偶然に正規表現の弱点を突く長い文字列が含まれる場合。 | `collect` コマンドがハングし、システム全体の利用が不能になる。 |
| **相対表現の曖昧性** | **中** | 「明日」「今週」などの単語を、基準日なしに抽出・保存した場合。 | 数日後にその TODO を見た際、期限がいつだったのか特定できなくなる。 |
| **DB 肥大化** | **低〜中** | 非常に短いキーワード（例：「する」「確認」のみ）を TODO ルールにした場合。 | `inferences` テーブルが数万件規模になり、検索パフォーマンスが低下する。 |
| **メンテ不能な正規表現** | **中** | 複数の意図が混ざった1つの巨大な正規表現を作成した場合。 | バグ修正やルール追加が困難になり、最終的に誰も触れなくなる。 |
| **再抽出の欠如** | **中** | ルール変更後に `re-scan` などの仕組みがない場合。 | 古いメモが新しいルールで見つからず、情報が埋もれる。 |

## 5. 進行を止めるべき論点
- **「いつ」をどう扱うか**: 「明日」などの相対時間を抽出する際、チャンクのタイムスタンプ（`block_timestamp`）やファイル更新日を基準日としてメタデータに含める設計が固まっているか？ これがないと、抽出した意味が半分失われる。

## 6. 最低限必要なガード
1. **正規表現のタイムアウト/制限**: Python の `re` モジュールは標準でタイムアウトを持たないため、極端に長い文字列を処理対象から外すか、正規表現を単純なキーワードの組み合わせに留める（複雑な後方参照などを避ける）。
2. **メタデータへの基準日保存**: `inferences` テーブルに、抽出時の基準日（またはチャンクの時刻）を必ず紐付ける。できれば、相対表現を絶対日時に（ルールベースで可能な範囲で）変換して保存する。
3. **カテゴリの明確化とテストケースの整備**: 各カテゴリ（TODO, DEADLINE, PENDING等）ごとに、期待する入力と「これは検知してはいけない」という反例（Negative Case）をテストコードで管理する。
4. **一括再抽出コマンドの検討**: `kuroko collect --re-inference` のような、既存の `chunks` テーブルに対してルールを再適用するインターフェースを用意する。

## 7. planning工程へ反映すべき修正
- **Schema 変更**: `inferences` テーブルに `inference_date` (抽出時の基準日) または `inference_meta` (JSON) で基準日や絶対日付変換結果を保持できるようにする。
- **InferenceEngine のモジュール化**: 正規表現をハードコードせず、リストや辞書で管理し、各パターンの説明（コメント）を必須とする。
- **処理フローの分離**: `memo_collector.py` に直接ロジックを書かず、`InferenceEngine` を独立したクラスとして実装し、単体テストを容易にする。
- **量制限**: 1つのチャンクから抽出できる `inferences` の上限数を設け、異常な増殖を防ぐ。
