# Implementation Notes - Issue #22: ルールベースによるTODO/期限等の抽出

## 実施内容

### 1. DBスキーマの更新 (`kuroko_core/db.py`)
- `inferences` テーブルを新規作成しました。
- `chunk_id` (chunks.id への外部キー) と `inference_type` にインデックスを付与しました。
- これにより、ルールベースによる抽出結果（推論）を構造化して保存・高速に検索できるようになりました。

### 2. 抽出エンジンの実装 (`kuroko/inference.py`)
- `InferenceEngine` クラスを実装しました。以下の 5 カテゴリの抽出に対応しています。
    - **TODO**: チェックボックス (`[ ]`)、`TODO:` プレフィックス、および特定の動詞（`送る`, `確認`, `実装` など）によるマッチング。
    - **DEADLINE**: `明日`, `明後日`, `明々後日`, `来週`, `再来週`, `来月` などの相対表現、および `M/D` 形式の絶対日時、`期限:` プレフィックス。
    - **PENDING**: `待ち`, `保留`, `確認中` などのキーワード。
    - **MEETING**: `会議`, `MTG`, `打ち合わせ`, `調整` などのキーワード。
    - **COMPARE**: `案1`, `比較`, `検討`, `メリット` などのキーワード。
- **相対日付の解決**:
    - 抽出時に基準日 (`base_date`) を渡すことで、相対表現を `YYYY-MM-DD` 形式の絶対日時に変換し、メタデータ (`resolved_date`) として保存します。
    - `明後日`, `明々後日`, `再来週` などのバリエーションを拡充しました。
    - チャンクに `block_timestamp` がある場合はそれを基準日とし、ない場合は現在時刻を基準とします。

### 3. 収集フローへの統合 (`kuroko/memo_collector.py`)
- `collect_memo` 関数を修正し、チャンクの保存・更新が行われた直後に `save_inferences` を呼び出すようにしました。
- 各チャンクについて、既存の推論結果を一度削除してから、最新のルールに基づいて再抽出・保存を行います。

### 4. CLIオプションの追加 (`kuroko/cli.py`)
- `kuroko collect memo --re-inference` オプションを追加しました。
- ルールを改善・変更した際、既存の全チャンクに対して一括で再抽出を行うことが可能です。

## 変更ファイル
- `kuroko_core/db.py`: `inferences` テーブル定義の追加。
- `kuroko/inference.py`: 抽出エンジン本体の実装（新規作成）。
- `kuroko/memo_collector.py`: 収集フローへの統合および再抽出ロジックの実装。
- `kuroko/cli.py`: `--re-inference` コマンドライン引数の追加。
- `tests/test_inference.py`: 抽出エンジンの単体テスト（新規作成）。

## 検証結果
- `pytest tests/test_inference.py` で抽出エンジンの単体テストが全件パスすることを確認しました。
- サンプルの `memo.md` を用いて、TODO、期限（絶対・相対）、会議、比較検討が意図通りに抽出され、`test.db` の `inferences` テーブルに正しく保存されることを確認しました。
- `--re-inference` オプションにより、既存データに対しても再抽出が行われることを確認しました。

## 未対応事項 / 次工程への申し送り
- 日付解決は `明後日`, `再来週` など主要なものを拡充しましたが、`3日後`, `再来月` などのより複雑なパターンが必要な場合は適宜追加が必要です。
- `shinko` (LLM層) では、これらの `inferences` をヒントとして利用するように実装を拡張する必要があります。
- 1つの行から重複して同じタイプが抽出されないようにしていますが、異なるタイプ（例: TODO と MEETING）が同時に抽出されることは許容しています。
