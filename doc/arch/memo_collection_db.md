# Architecture & Database Design: Memo Collection (Phase 1)

## 1. 概要
`kuroko` におけるメモ収集基盤の初期設計ドキュメントです。プロジェクト内の `memo.md` を収集し、検索やLLM解析のソースとして永続化するためのデータ構造と、設計上の重要な決定事項を記録します。

## 2. システム構成
3層アーキテクチャに基づき、関心を分離しています。

1.  **UI Layer (`kuroko/cli.py`)**: ユーザー入力を受け取り、設定をロードし、コレクターを実行する。
2.  **Logic Layer (`kuroko/memo_collector.py`)**: ファイルシステムを走査し、ハッシュ計算とDBへの登録・更新判断を行う。
3.  **Core Layer (`kuroko_core/db.py`)**: SQLite接続の管理、スキーマ定義、原子的なデータ操作（将来的に拡張）を担う。

## 3. データベース設計 (SQLite)

### テーブル: `source_texts`
プロジェクトから収集された原文（Raw Text）を格納する中心的なテーブルです。

| カラム名 | 型 | 制約 | 説明 |
| :--- | :--- | :--- | :--- |
| `id` | INTEGER | PRIMARY KEY | 内部管理用ID |
| `source_type` | TEXT | NOT NULL | ソースの種類（現在は "memo" 固定） |
| `path` | TEXT | NOT NULL UNIQUE | ファイルの絶対パス。1つのパスにつき1レコード。 |
| `directory_context` | TEXT | - | ファイルの親ディレクトリ名など、階層情報を保持。 |
| `raw_text` | TEXT | NOT NULL | ファイルから読み取った生のテキスト内容。 |
| `file_hash` | TEXT | NOT NULL | 内容の SHA-256 ハッシュ値。重複インポート防止に使用。 |
| `updated_at` | DATETIME | DEFAULT CURRENT_TIMESTAMP | **内容が変更された**最終時刻。 |
| `imported_at` | DATETIME | DEFAULT CURRENT_TIMESTAMP | **最初にインポートされた**時刻。更新時も維持。 |

### インデックス
- `idx_source_texts_hash` (`file_hash`): ハッシュによる高速な重複存在チェック用。
- `path` (UNIQUE INDEX): パスによる重複登録の防止と、更新時のレコード特定用。

## 4. 設計判断 (Design Decisions)

### 4.1. ストレージに SQLite を選択した理由
- **ゼロ構成**: サーバーのセットアップが不要で、ユーザーのホームディレクトリ配下に単一ファイルとして配置できる。
- **ACID特性**: 収集プロセスが途中で中断しても、データの一貫性を保ちやすい。
- **将来性**: 後のフェーズで `fts5` モジュールを用いた全文検索や、ベクトル検索プラグインの導入が容易。

### 4.2. パスベース vs ハッシュベースの重複排除
本実装では、以下の2段階のチェックを採用しています。

1.  **内容の重複防止 (Global Hash Check)**: 
    - 内容（ハッシュ）がすでにDB内に存在する場合、それがどのパスにあってもインポートをスキップする。
    - **理由**: 同一内容のメモを複数箇所にコピーした場合に、冗長なレコード（および将来のLLMコスト）を避けるため。
2.  **パスの同一性 (Path Integrity)**:
    - パスに `UNIQUE` 制約を付与。
    - **理由**: 同一ファイルが編集された場合、新規登録ではなく既存レコードを「更新」として扱うため。

### 4.3. 時刻管理の分離 (`imported_at` vs `updated_at`)
- `imported_at` はレコード作成時のみセットされ、以降の更新では変更しません。
- `updated_at` は内容が変更されるたびに更新されます。
- **判断理由**: 「いつからそのメモを追跡し始めたか（歴史）」と「最新の内容はいつか（鮮度）」を区別できるようにするため。

## 5. 将来の拡張性への考慮
- **`source_texts` の拡張**: 今後、要約やカテゴリ（LSP的なシンボル情報など）を保存する際、このテーブルに外部キーを貼るか、カラムを追加することで容易に拡張可能です。
- **ベクトル検索**: `file_hash` があることで、内容が変わっていない場合は埋め込み（Embedding）の再計算をスキップするロジックを容易に実装できます。
- **マルチソース**: `source_type` を持たせることで、将来的に `memo.md` 以外（例: slackの履歴、git commitログ）を収集する場合も同じテーブル構造を流用可能です。
